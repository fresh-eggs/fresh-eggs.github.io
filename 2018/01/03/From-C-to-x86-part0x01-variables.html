From C To X86 : Part0x01 Variables

January 03, 2018

Reading time ~6 minutes

Welcome back for another action packed installment of x86 assembly for C programmers!

We’ll be focusing on all things varibles in this article. We’ll take a look at the two major allocation scenarios consisting of memory on the stack and memory on the heap.



The C code

#include <stdio.h>

int heap(){
	int *heap_addr = (int *) malloc(2 * sizeof(int));
	printf("this is the heap address we got from malloc: 0x%d\n", heap_addr);
	*heap_addr = 5; //simple assignment.
	heap_addr[0] = 6; //array style notation to do the same as the last assignment
	return 0;
}

int stack(){
	int x = 1;
	int y = 2;
	x = 3;
	x = y;
	return 0;
}

int main(){
	heap();
	stack();
	return 0;
}

Laid out above are a set of functions that will allocate variables in different contexts.

With that, how about we bust up some assembly?



Assembly breakdown

main()

The scope of the variable below is inside the main() function. Since we assigned it a value directly without using dynamic memory given to us our weapon of choice (calloc, malloc, etc…), this varibale will live on the stack.

int main_x = 8;

This is made evident by the following lines. (note these are not sequential in the actual binary, I ordered them for the example):

; var int local_10h @ ebp-0x10
0x08048544      83ec14         sub esp, 0x14
0x08048552      c745f0080000.  mov dword [local_10h], 8

Above is a pretty straightforward allocation and assignment procedure for a variable sotred on the stack.

We get a hint that it is on the stack from the first line, a comment from radare2 telling us it labled the space at ebp-0x10 as a local varibale of type int.

Next we substract from the ESP to make room for local variables, followed up by a simple mov instruction to litteraly move the value into the location pointed to by the local_10h address.

Here is a diagram to help, imagine if there was only one local var and replace the name with local_10h:

Note: Not all local stack variables will be used to represent things you explicitly created.

An important take away from this section is to recognize that a local stack variable is nothing more than an offset from the EBP (base pointer).

As the stack shrinks when it grows, you should recognize that the stack variables will be held in the address space between the ESP and the EBP.


stack()

In the case of our “stack” function, we’ll break it down into two parts.

Below are the simple allocations, assign a value into the location (memory address) pointed to by local_8h.

int x = 1;
int y = 2;
x = 3;

0x080484e0      83ec10         sub esp, 0x10
0x080484e3      c745f8010000.  mov dword [local_8h], 1
0x080484ea      c745fc020000.  mov dword [local_4h], 2 
0x080484f1      c745f8030000.  mov dword [local_8h], 3

We’re local_8h not wrapped in [], it would be like we are saying “replace the address held in local_8h with 1”.

Notice the substitution of the stack pointer to make way for our variables X and Y.

Also, you might be asking “why are there two extra bytes subtracted from the ESP?”. For now, consider that compiler magic.


Next up is an assignment operation with a little more beef to it:

x = y;
return 0;

First two lines below are responsible for moving the value held at local_4h into the EAX, followed by again, moving the value we put in EAX into the location local_8h points to.

The rest simply helps with returning zero.

0x080484f8      8b45fc         mov eax, dword [local_4h]
0x080484fb      8945f8         mov dword [local_8h], eax
0x080484fe      b800000000     mov eax, 0
0x08048503      c9             leave
0x08048504      c3             ret



heap()

Now we’ll take a look at a different style of memory allocation entirely, with the help of our friend the heap.

First is the simple allocation of the starting address to the memory region we were given:

int *heap_addr = (int *) malloc(2 * sizeof(int));

This simple assignment translates to the following assembly operations:

0x080484a4      6a08           push 8                      ; 8                                           
0x080484a6      e8c5feffff     call sym.imp.malloc         ;[2] ;  void *malloc(size_t size)
0x080484ae      8945f4         mov dword [local_ch], eax

The 8 bytes being pushed onto the stack represent our request for 2 times the size of an int (32-bit so 4 bytes(int) times two).

Also remember that parameters are generally passed on the stack in a 32 bit environment

Next up is moving the value in EAX (the return value from malloc) into where lable local_ch points to. In our case, this is heap_addr(local_ch or ebp-0xc).

An important take away from this section is the [var] notation in assembly is similar to de-refferencing a pointer in C.


Now that we have stored our address in the pointer at local_ch, we can make use of this address for assignments:

*heap_addr = 5; //simple assignment.
heap_addr[0] = 6; //array style notation to do the same as the last assignment
return 0;

Translated to assembly, we can see that we move the contents of the variable local_ch into the EAX (heap address). Once in the EAX, we de-refference the address to place 5 in it.

Also note that regardless of C syntax, both assignment operations translate to the same assembly.

0x080484c4      8b45f4         mov eax, dword [local_ch]
0x080484c7      c70005000000   mov dword [eax], 5
0x080484cd      8b45f4         mov eax, dword [local_ch]
0x080484d0      c70006000000   mov dword [eax], 6
0x080484d6      b800000000     mov eax, 0
0x080484db      c9             leave
0x080484dc      c3             ret
